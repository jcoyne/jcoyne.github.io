Forms and Presenters and Indexers! Oh my!

Form

  • Represent the edit page.
  • Interface between Controller - View
  • Input for the update action
  • Could be use to autogenerate the form

Presenter

  • Represent the show page.
  • Interface between Controller - View
  • Could wrap a Fedora model, but in Hydra we get better perfomance if we wrap a SolrDocument
  • Could be use to autogenerate part of the view page
  • Can replace Helpers

Indexer

  • New part of Hydra
  • replaces overriding 
    to_solr
  • Put a Fedora model into Solr
  • Companion to Presenter. Indexer - Map to solr. Presenter - Map from solr.

Why?

  • MVC - Works fine for small apps. Bigger apps need more organization
  • MVC - Works best with a single data store. Hydra has two, Fedora and Solr.
  • Provides a pathway to customization
  • Smaller classes, single responsibility
  • Faster tests because we're isolating code better
